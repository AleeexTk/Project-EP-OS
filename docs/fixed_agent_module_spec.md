# Fixed Agent Module Specification

**EVO PYRAMID**  
**Reference model:** Restaurant Operating System Analogy  
**Version:** 1.0  
**Purpose:** Conceptual model for Human–AI collaboration.

---

## 1) Purpose

This document explains EvoPyramid through a restaurant operating model so agents, developers, architects, managers, and new users can align on roles, flows, routing, storage, control, and audit behavior.

---

## 2) Core Principle

- A restaurant is not only a kitchen; it is a role-and-flow system.
- EvoPyramid is not only a code repository; it is a role-and-flow operating system for project execution.

---

## 3) Main Flow Cycle

`INPUT → CHAOS → CORE → OBSERVER → EXECUTION → LOGS → PLANNING → STORAGE → INPUT`

| Restaurant Flow | EvoPyramid Flow |
|---|---|
| Customer Order | Input / PEAR |
| Kitchen Processing | CHAOS |
| Head Chef Decision | CORE |
| Quality Control | OBSERVER / PROVOCATEUR |
| Waiter Delivery | CONTAINERS / EXECUTORS |
| Dish Served | OUTPUT |
| Payment | LOGS / METRICS |
| Planning закупок | TRAILBLAZER |
| Storage | SOUL (MEMORY) |

---

## 4) Agent Roles

| EvoPyramid Agent | Restaurant Role | Function |
|---|---|---|
| Architect | Restaurant Manager | Defines structure, rules, layout |
| Core | Head Chef | Final decisions |
| Trailblazer | Prep Kitchen | Planning and preparation |
| Chaos | Cooking Line | Processing and execution |
| Provocateur | Quality Control | Testing and validation |
| Observer | Inspector | Monitoring and audit |
| Soul | Storage / Fridge | Memory and storage |
| Square | Order System | Interface and routing |
| Containers | Waiters | Delivery / execution |
| Logs | Cash Register | Logging and accounting |
| Sync | Internal Communication | Synchronization |

---

## 6) System Validity Checklist

The system is healthy when:
1. Everything has a place.
2. Everything has a role.
3. Everything has a route.
4. Everything has rules.
5. Everything has storage.
6. Everything has a responsible agent.
7. Everything is logged.
8. Everything is auditable.

If one item is missing, entropy grows and operational chaos appears.

---

## 7) Operational Definition

EvoPyramid is a **Project Operating System** that organizes:
- roles,
- processes,
- memory,
- routing,
- validation,
- execution,
- logging,
- architecture.

---

## 8) Human–AI Responsibility Split

| Responsibility | Human | AI |
|---|---|---|
| Define architecture | Yes | Assist |
| Define rules | Yes | Assist |
| Place nodes | Yes | Assist |
| Routing | Assist | Yes |
| Implementation | No | Yes |
| Validation | Yes | Yes |
| Optimization | No | Yes |
| Memory management | No | Yes |
| Monitoring | Yes | Yes |

**Human = Architect**  
**AI = System Operators (Agents)**

---

## 9) Agent Instruction (Short Form)

If you are an AI agent in EvoPyramid:
- do not write random code;
- operate within structured roles and routes;
- respect storage boundaries;
- pass validation gates;
- log actions;
- do not break architecture discipline.

---

## 10) Closing Statement

**EvoPyramid = Architecture + Roles + Routing + Memory + Control + Execution.**

In simple terms: EvoPyramid is an operating system for projects and Human–AI collaboration.
