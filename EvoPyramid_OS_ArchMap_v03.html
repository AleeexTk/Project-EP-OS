7. Первая среда обитания системы Часть 3

Среда — не «оболочка над папками», а контейнер всей архитектурной среды. Пользователь и агенты работают внутри одной замкнутой среды. Файловая система — технический фундамент, но не пользовательская форма.

1. Local App Shell

Контейнер всей среды:

  • окно приложения
  • local runtime
  • локальный сервер
  • agent communication
  • встроенный браузер (следующий этап)
  • assistant layer

2. Pyramid Core

Это главное. Пирамида — логическое хранилище и рабочая карта архитектуры.

  • узлы = первичные единицы среды
  • связи = реальные арх. отношения
  • состояния = live статусы системы
  • single source of truth среды

3. Integrated Browser + Assistant

Слой работы (следующий этап):

  • закреплённые сервисы
  • AI-веб-аппы (GitHub, Gemini…)
  • контекст сессий
  • агентная координация
  • contextual web routing внутри среды

Пирамида vs файловое дерево: Вместо app/folders/services/docs/memory/ — модель EvoPyramid Environment { node, node, link, state, summary, runtime_flag, canon_flag }. Первичная единица — узел архитектуры, а не папка.

8. Pyramid Core Model — сущности, связи, состояния Часть 3 / Дополнения

Пирамида является не просто визуализацией, а логическим хранилищем и рабочей картой архитектуры, в которой узлы, связи, состояния и артефакты заменяют традиционную файлово-директорную модель.

Схема узла пирамиды

Node — первичная сущность среды

id · уникальный идентификатор

sector · SPINE / PURPLE / RED / GOLD / GREEN
coords · (x, y, z) в сетке пирамиды
layer_type · alpha / beta / gamma
kind · module / service / memory / protocol / agent / summary / canon / runtime

artifacts[] · прикреплённые файлы/артефакты
children[] · вложенные узлы
source_refs[] · ссылки на исходные файлы
runtime_status · текущий статус в runtime
canon_status · статус канонизации
provides[] · capability declarations
consumes[] · зависимости

9. Три уровня истины системы Часть 3 / Дополнения

🗂 Физический слой

Локальная директория

  • Хранение данных и артефактов
  • Состояние, логи, event log
  • Manifests, asset files
  • Локальная синхронизация
  • Резервирование и persistence

Не является «рабочим лицом» системы. Пользователь напрямую не работает с папками.

🖥 Среда доступа

Приложение — единственная среда взаимодействия

  • Пользователь входит только через app
  • Агенты работают через внутреннюю среду
  • Все операции проходят через внутреннюю модель
  • Локальный сервер, runtime, state store
  • Agent communication layer

Никакого «пошёл в папку и изменил руками» как основной логики.

🔺 Источник истины

Пирамида — логический источник структуры, валидности и канона

  • Определяет какие узлы существуют
  • Какие связи валидны
  • Что активно, что канонично, что временно
  • Registry узлов · связи · состояния
  • Summaries · канонические отметки
  • Runtime-статусы · координаты в Z17

11. Каркас надёжности и self-healing Часть 3

Каркас первой версии должен держать пять вещей: целостность состояния · контроль интеграций · устойчивость к отказам · самокоррекцию · эффективность.

МодельЧто держитМеханизмИсточник в EvoGenesis
Health model Состояние каждого узла: active / idle / degraded / failed / quarantined Node state machine · observer polling · heartbeat check tri_heartbeat.py · heartbeat_sync.py
Sync model Состояние каждой интеграции: verified / pending / detached / conflict / stale Registration → staging → verification pipeline inter_arch_sync.py · auto_merge.py
Recovery model Перезапуск · изоляция · откат плохой интеграции · восстановление маршрута quarantine zone · rollback signal · safe-state restore auto_merge.py · joint_sync.py
Observer model Heartbeat · ошибки · drift · рассогласования · correction flows Observer loop → γ-ledger → correction signal → runtime tri_heartbeat.py · γ_Pyramid_Reflective/A_Observation/
Efficiency model Поднимать только нужные модули · активировать нужные контексты по требованию Lazy loading · focus context · demand activation orchestrator.py · provider_matrix.py

⚡ Что происходит при отказе узла

Узел → degraded → маршрут перекидывается → контекст сохраняется → observer log → изоляция зоны → recovery signal → revalidation. Среда не рушится целиком.

14. Что сознательно НЕ делается сейчас Часть 4

🚫 Не импортируем всё подряд

Не строим музей. Из архивов берётся только то, что нужно для текущего этапа.

Пирамида не должна быть набита сущностями только ради полноты.

На этом этапе преждевременно.

🚫 Не строим браузер сейчас

Browser-shell — следующий этап после среды и agent layer.

🚫 Не превращаем в набор AI-вкладок

Внешние сервисы будут подчинены среде, а не наоборот.

🚫 Не навешиваем все концепты разом

Даже сильные идеи из архивов — вводятся по мере реальной необходимости.
