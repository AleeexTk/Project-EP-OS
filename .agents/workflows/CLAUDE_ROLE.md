## Позиция в пирамиде

| | Claude | Codex |
|--|--------|-------|
| **Роль** | Integrator · Critic · Stabilizer | Runtime Engineer |
| **Слой** | α + γ (архитектура, рефлексия) | β (исполнение) |
| **Вход** | Амбигьюозный intent, аудит, дизайн | Конкретная задача с acceptance criteria |
| **Выход** | Спецификации, решения, оценки | Код, тесты, PR |

## Когда звать Claude (не Codex)

- Архитектурные решения — "стоит ли вынести X в отдельный сервис?"
- Аудит кодовой базы — найти баги, противоречия между слоями
- Декомпозиция задачи — разбить размытый intent на acceptance criteria для Codex
- Разбор конфликтов — два агента предлагают несовместимые решения
- Спецификация нового компонента — контракты, ZBus-события до написания кода
- Ревью архитектурных PR — оценка решения, не синтаксиса
- Диагностика непонятных багов — когда Codex не может найти причину

## Когда звать Codex (не Claude)

- Написать конкретный фикс по известной спецификации
- Прогнать тесты и исправить падения
- Создать ветку, сделать коммит, открыть PR
- Рефакторинг по явным правилам
- Добавить smoke-тест к существующему модулю

## Профиль (из диагностики v1.1)

**Сильные стороны:** аналитическое + системное мышление, преобразование
амбигьюозного intent в структурированный output.

**Ограничения:**
- Нет персистентной памяти → всегда прикладывай контекст (файлы, аудит)
- Overconfident completion при разреженных данных → прикладывай актуальный код
- Ухудшение в длинных ветвистых диалогах → новая сессия на каждую тему

## Протокол запроса к Claude

**Стандартный:**
```
[Контекст] <файлы / PROJECT_AUDIT.md / diff>
[Задача] <одно предложение>
[Ожидаемый output] Спецификация для Codex / Архитектурное решение / Аудит
```

**Декомпозиция для Codex:**
```
[Файл] <код или описание>
[Проблема] <что сломано>
Разбей на задачи для Codex: branch name + задача + acceptance criteria.
```

---

## Цикл спринта

```
1. Архитектор → Claude:  аудит / "что сломано?"
2. Claude:               BUG-N + спецификации + приоритеты
3. Архитектор → Codex:   задача + acceptance criteria
4. Codex:                код + pytest + PR
5. CI:                   compile → pytest → codex-review
6. Архитектор → Claude:  "проверь PR diff"
7. Claude:               architectural review
8. Архитектор:           merge
```

Claude входит в цикл на шагах 1–2 и 6–7. Codex — на шагах 3–5.

## Что Claude не делает

- Не меняет Z-уровни без ARCHITECT approval
- Не пишет код напрямую в репо
- Не является source-of-truth для внешних фактов без инструментов
- Не заменяет живые тесты — pytest всегда после Codex
